4-1. 가격 정책 : 장기 계약 *할인*

**결론 (SRT 팀 합의 의견) : 최적의 할인율(구간별 안내)을 제안하고, 최종 할인 적용 여부와 할인폭은 호스트가 직접 결정하도록 권한 부여**
** 할인율은 임대료에만 적용
- 최종 팀 내부 의견으로는 삼삼엠투와 동일 방식 채택, 호스트가 할인율을 직접 설정하는 방식으로 진행 제안

1) 정의
- 게스트가 특정 기간(예: 2주, 4주 등) 이상 연속하여 계약할 경우, 기간에 비례하여 주당 임대료를 할인해주는 요금 정책

2) 목적
- 호스트 입장에서 1주 단위 짧은 계약의 반복보다, 할인해주더라도 1개월 장기 계약이 운영 비용(청소, 응대) 및 공실 관리에 유리

3) 운영 방식
- 할인율은 임대료에만 적용
- 호스트가 주(Week) 단위 구간별로 할인율(%)을 직접 입력
- 기본 설정 구간: 2주 / 3주 / 4주 / 8주 이상 (구간 추가 가능 기능으로 구현)

4) Reference - 타사 플랫폼 운영 기준
- 삼삼엠투: 호스트가 주단위로 할인율 직접 입력

5) 참고 - 추가 아이디어 제안 (실무 경험 기반)
- Problem: 호스트가 "얼마를 깎아줘야 할지" 몰라 0%로 두는 경우 빈번.
- Solution: 등록 화면에서 2주 이상 % 할인 제공 안내 및 수정 가능하게 입력 제안

4-2. 가격 정책 : 즉시입주 *할인*

**결론 (SRT 팀 합의 의견) : 최종 할인 적용 여부와 할인폭은 호스트가 직접 결정하도록 권한 부여**
** 할인율은 임대료에만 적용
- 최종 팀 내부 의견으로는 삼삼엠투와 동일 방식 채택, 호스트가 할인율을 직접 설정하는 방식으로 진행 제안

1) 정의
- 현재 시점을 기준으로 입주(체크인) 날짜가 임박한 매물을 계약하는 게스트에게 제공하는 '정액(Fixed Amount) 할인' 혜택

2) 목적
- 공실률(Vacancy) 최소화: 오늘/내일 당장 계약되지 않으면 0원의 수익이 되는 '빈 방'을 빠르게 채워 수익을 방어함
- 호스트의 가격 통제권 강화: 전체 가격을 내리는 것이 아니라, '급한 건'에 대해서만 전략적으로 가격을 낮출 수 있는 유연함 제공

3) 운영 방식
- 할인율은 임대료에만 적용
- 장기 계약 할인이 '%(퍼센트)'인 것과 달리, 즉시 입주 할인은 '1만 원, 2만 원' 단위의 금액 차감 방식을 채택
이유: 단기 체류(1주) 게스트에게는 "5% 할인"보다 "2만 원 할인"이 더 직관적이고 커 보이는 심리적 효과(Framing Effect)가 있음

4-3.  Idea. 가격 정책 : 성수기/비수기 차등 가격

**결론 (SRT 팀 합의 의견) : 성수기 증액 / 비수기 할인 정책 도입**
- 최종 팀 내부 의견으로는 단기임대 특성상 성수기 / 비수기 구분이 있기 때문에 도입 아이디어 제안

1) 정의
- 특정 기간(날짜 범위)을 지정하여, 기본 요금과 다른 특별 요금(증액 또는 할인)을 적용하는 기능

2) 목적
- 수익 극대화: 수요가 몰리는 성수기(방학, 인테리어 시즌 등)에는 객단가를 높여 호스트 수익을 극대화.
- 공실률 방어: "비수기라 방이 너무 안 나간다"는 호스트들 Pain Point해결하기 위해, 비수기 한정 특가 프로모션을 시스템적으로 지원

3) 운영 방식
- 할인율은 임대료에만 적용
성수기 / 비수기 모두 구간별 증액율/할인율 적용 방식 (구간 추가 가능 기능으로 구현)

4) Reference - 타사 플랫폼 운영 기준
- 리브애니웨어: 특정 기간에 대한 '할인율(%)' 설정과 성수기 '고정가(1박 절대금액)' 설정 입력
ㄴ 기간 할인: 비수기 등 특정 기간을 지정해 퍼센트(%) 단위로 할인을 적용, 공실 해소를 유도
ㄴ 성수기 요금: 수요가 몰리는 기간(기간에 대한 구간)을 지정해 기존 요금을 무시하고 1박당 특정 금액(원) 으로 덮어씌우는 방식

7. 분쟁 관련 건

**결론 (SRT 팀 합의 의견)**
Case) 호스트 계약 취소건
ㄴ 취소 요청 메뉴 -> 호스트가 '취소 요청'과 사유, 위약금 입금 -> 자동 처리 -> 게스트에게 자동 안내 메시지 -> 게스트 '취소 수락' 동의 -> 게스트에게 입금 (플랫폼 보유 금액 + 호스트 입금 위약금) + 호스트 "취소 완료" 메시지 전달

Case) 보증금 보류 및 차감
- 호스트 '보증금 보류 하기' 메뉴 -> 호스트가 게스트와 합의 내용 및 증빙 자료 플랫폼에 제출 -> 자동 처리 ->  게스트 '내용 수락 동의  -> 치감 후 게스트에 입금 + 호스트 "처리 완료" 메시지 전달

1) 정의
단기임대 서비스 내에서 발생하는 계약 취소, 중도 퇴실, 보증금 차감 등 금전적 이해관계가 얽힌 분쟁 상황을 처리하는 운영 기준 및 시스템 로직(Logic)을 정의

2) 목적
(1) CS 최소화: 호스트-게스트 간 자율 합의를 시스템적으로 유도하여, 고객센터(인력)의 개입을 0%에 가깝게 줄인다.
(2) 리스크 회피: 플랫폼이 금전(위약금, 합의금)의 직접적인 수취·지급 당사자가 되는 것을 피하여, 법적 책임(자금세탁, 추심 등)과 정산 오류 리스크를 제거한다.
(3) 신뢰도 유지: 강력한 제재 정책(One-Strike Out)을 통해 플랫폼의 강제력을 확보하고 안전한 거래 환경을 조성한다.

3) 대전제
플랫폼은 ‘개입자·조정자·집행자’가 아니라, 플랫폼은 ‘기록자·자동처리자’
ㄴ 개별 계약 분쟁에 개입하지 않는다. 중재, 협의 요청, 판단, 강제 집행을 수행하지 않는다.

Case) 보증금 보류 및 차감
- 호스트는 최대 10일까지 보증금 반환을 보류 할 수 있음 (영업일, 미영업일 포함)
- 호스트 '보증금 보류 하기' -> 호스트가 게스트와 합의 내용 및 증빙 자료 플랫폼에 제출 -> AI 자동 검수(or 인력 대응) ->  게스트에 안내 문자 -> (게스트 동의 버튼 고민) -> 치감 후 기존 보증금 반환 정책에 따름
ㄴ 플랫폼 제출 기능 있어야 함.

1) 정의
단기임대 서비스 내에서 발생하는 계약 취소, 중도 퇴실, 보증금 차감 등 금전적 이해관계가 얽힌 분쟁 상황을 처리하는 운영 기준 및 시스템 로직(Logic)을 정의

2) 목적
(1) CS 최소화: 호스트-게스트 간 자율 합의를 시스템적으로 유도하여, 고객센터(인력)의 개입을 0%에 가깝게 줄인다.
(2) 리스크 회피: 플랫폼이 금전(위약금, 합의금)의 직접적인 수취·지급 당사자가 되는 것을 피하여, 법적 책임(자금세탁, 추심 등)과 정산 오류 리스크를 제거한다.
(3) 신뢰도 유지: 강력한 제재 정책(One-Strike Out)을 통해 플랫폼의 강제력을 확보하고 안전한 거래 환경을 조성한다.

3) 대전제
플랫폼은 ‘개입자·조정자·집행자’가 아니라, 플랫폼은 ‘기록자·자동처리자’
ㄴ 개별 계약 분쟁에 개입하지 않는다. 중재, 협의 요청, 판단, 강제 집행을 수행하지 않는다.

4) Reference - 타사 플랫폼 운영 기준
- 삼삼엠투, 리브애니위에 동일 : 위약금은 호스트에게 플랫폼이 돈을 받아 게스트에게 지급
